# Reference-data schema

Captured Fortran outputs live under this directory:

```
tests/reference/
├── SCHEMA.md                # this file
├── sweep/                   # baseline 12-point convergence sweep (NetCDF)
│   └── mam_dt<DT>_ndt<N>.nc
├── indices/                 # one-time integer index tables (.npz)
│   └── reference.npz
├── per_process/             # instrumented per-microphysics-call dumps (.npz)
│   ├── calcsize_before.npz
│   ├── calcsize_after.npz
│   ├── wateruptake_before.npz
│   ├── wateruptake_after.npz
│   ├── amicphys_before.npz
│   └── amicphys_after.npz
├── polysvp/                 # standalone polysvp T-sweep
│   └── reference.npz
└── qsat/                    # standalone qsat (T, p)-grid
    └── reference.npz
```

Both formats are regenerated by `python scripts/capture_reference.py` (see [`docs/REFERENCE_BUILD.md`](../../docs/REFERENCE_BUILD.md) for prerequisites). They are committed so the test suite can run without a Fortran toolchain on every host.

## Baseline sweep — `tests/reference/sweep/*.nc`

Twelve NetCDF files, one per timestep count in the canonical sweep `(1, 2, 4, 9, 18, 30, 60, 120, 180, 360, 900, 1800)` over a fixed 1800 s integration window. Naming: `mam_dt<DT>_ndt<N>.nc` where `DT = 1800/N`.

Written by the unmodified `driver.F90` (no instrumentation overlay applied). Variables, dimensions, and metadata are defined by the upstream Fortran's NetCDF output code — inspect with `ncdump -h tests/reference/sweep/mam_dt1800_ndt1.nc`. The MAM4-JAX port must reproduce these files to within ADR-003's `1e-6` relative error at every timestep count (Milestone 5).

## Integer index tables — `tests/reference/indices/reference.npz`

Captured **once at init** by `dump_indices()` in `scripts/patches/mam4_dump_state.F90`, called by the patched `driver.F90` immediately before the `main_time_loop`. Records the values `modal_aero_initialize_data.F90` assigns to the runtime index arrays after name-lookup against the constituent list.

Arrays (all `int32` unless noted):

| Key | Shape | Meaning |
| --- | --- | --- |
| `ntot_amode`, `ntot_aspectype`, `maxd_aspectype` | scalar | Compile-time mode/species counts. |
| `numptr_amode`, `numptrcw_amode` | `(ntot_amode,)` | **0-based** pcnst index of the (interstitial / cloud-borne) number tracer for each mode. |
| `nspec_amode` | `(ntot_amode,)` | Number of species in each mode. |
| `lspectype_amode` | `(ntot_amode, maxd_aspectype)` | **0-based** index into `specname_amode` for each (mode, slot). `-1` for unused slots. |
| `lmassptr_amode`, `lmassptrcw_amode` | `(ntot_amode, maxd_aspectype)` | **0-based** pcnst indices of the (interstitial / cloud-borne) mass tracer for each (mode, slot). `-1` for unused slots. |
| `modename_amode` | `(ntot_amode,)` strings | Canonical mode names. |
| `specname_amode` | `(ntot_aspectype,)` strings | Canonical species type names. |

Index conversion: the Fortran writes **1-based** indices with `0` for unused slots. `scripts/capture_reference.py` converts to **0-based** (subtract 1) with `-1` for unused. Indices in `mam4_jax/data.py` follow the same 0-based convention.

The hard-coded constants in `mam4_jax/data.py` (`NUMPTR_AMODE`, `LMASSPTR_AMODE`, etc.) must match this `.npz` — `tests/test_scaffolding.py::test_index_tables_match_npz_reference` enforces that.

## What the captured references actually exercise

Empirical note (added 2026-05-19, after the 60-step instrumented run was inspected): the box-model namelist + initial conditions do **not** drive the aerosol state out of normal operating ranges, so several Fortran code paths are dead in the captures. The most consequential examples:

- **`modal_aero_calcsize_sub`:** the number-tracer bounds-adjustment branches and the Aitken↔accumulation mode-transfer branches never trigger. All four mode-number tracers change by exactly zero between `calcsize_before` and `calcsize_after` across 60 timesteps; the only update calcsize makes is to `dgncur_a` (recomputed from updated mass + fixed number).
- **`modal_aero_amicphys` sub-processes** (gasaerexch, newnuc, coag, rename): only the outer `amicphys_before`/`amicphys_after` is captured. Individual sub-process contributions are not separable from this capture alone — a single-toggle re-run (e.g., `mdo_gasaerexch=1` only, others off) would be needed.

JAX ports of these processes are implemented in full (rule #6 — match Fortran), so they should be correct on inputs that *do* exercise the dead branches. But the .npz-based regression test cannot directly detect a bug in those branches with the current fixture. See `docs/DEFERRED.md` for resurface conditions.

## Per-process dumps — `tests/reference/per_process/*.npz`

Six NumPy `.npz` archives, one per (microphysics process, before/after) point inside `cambox_do_run`. Written by the instrumented build (`scripts/build_reference.sh --instrumented`) — the patch overlay inserts `call dump_snapshot('<tag>', ...)` at the six call sites described in ADR-012.

### Capture points

| `tag` | Inserted relative to driver.F90 | Captures |
| --- | --- | --- |
| `calcsize_before` | Before line 1118's `call modal_aero_calcsize_sub` | Inputs to calcsize |
| `calcsize_after` | After the apply-tendencies block at line 1155 | Outputs of calcsize, with `q` updated by the tendencies |
| `wateruptake_before` | Before line 1208's `call modal_aero_wateruptake_dr` | Inputs to wateruptake |
| `wateruptake_after` | After the post-call `unload_pbuf` | Outputs of wateruptake (wet diameter, wet density, aerosol water) |
| `amicphys_before` | Before line 1283's `call modal_aero_amicphys_intr` | Inputs to amicphys (interstitial state pre-VMR-conversion) |
| `amicphys_after` | Immediately after the amicphys call | Outputs of amicphys |

### Arrays in each `.npz`

Every `.npz` contains the same seven arrays. Leading axis is the number of times the hook fired during the run (= `nstep`); the per-record axes mirror the Fortran shapes.

| Key | dtype | Shape | Meaning |
| --- | --- | --- | --- |
| `istep` | `int32` | `(nstep,)` | 1-based timestep index, as written by Fortran |
| `q` | `float64` | `(nstep, ncol, pver, pcnst)` | Interstitial tracer mixing ratios (kg/kg-dry-air for aerosol species, mol/mol for gases) |
| `qqcw` | `float64` | `(nstep, ncol, pver, pcnst)` | Cloud-borne tracer mixing ratios |
| `dgncur_a` | `float64` | `(nstep, ncol, pver, ntot_amode)` | Current dry mode diameter (m) |
| `dgncur_awet` | `float64` | `(nstep, ncol, pver, ntot_amode)` | Current wet mode diameter (m) |
| `qaerwat` | `float64` | `(nstep, ncol, pver, ntot_amode)` | Aerosol water content per mode (kg/kg) |
| `wetdens` | `float64` | `(nstep, ncol, pver, ntot_amode)` | Wet aerosol density per mode (kg/m³) |

For the reference build config (`PCOLS=1`, `PVER=1`, `PCNST=35`, `NTOT_AMODE=4`), per-record sizes are `q,qqcw = 35 doubles each`, mode arrays `4 doubles each`. One record = 704 B in the underlying `.bin` (+4 B for the `istep` prefix).

### Note on `amicphys_*` and VMR conversion

`modal_aero_amicphys_intr` operates on **volume mixing ratios** (`vmr`, `vmrcw`), not mass mixing ratios (`q`, `qqcw`). The conversion happens at `driver.F90:1217-1228` between the wateruptake and amicphys calls. The dumps capture `q`/`qqcw` for layout consistency across the six tags; to compare against amicphys's internal VMR state, multiply by `mwdry/adv_mass(species_index)` (constants live in `chem_mods.F90` / `physconst.F90`). A future port that needs the VMR view directly can extend the schema or do the conversion in the loader.

## Reproducing the captures

```bash
# Baseline sweep (NetCDFs):
python scripts/capture_reference.py

# Per-process instrumented capture for a single timestep (default):
python scripts/capture_reference.py --mode instrumented

# Per-process capture at a longer integration:
python scripts/capture_reference.py --mode instrumented --nstep 60
```

The committed `per_process/*.npz` were generated with `--nstep 1` (single-step, 1800 s timestep). Re-running with the same flag should reproduce them bit-for-bit on a host with the same compiler/library versions; small differences across `gfortran` versions are expected.

## Binary `.bin` format (transient — for reference only)

The instrumented Fortran helper (`scripts/patches/mam4_dump_state.F90`) writes intermediate `mam4_dump_<tag>.bin` files into the run directory, which `capture_reference.py` parses into `.npz`. The `.bin` files are not committed and not part of the data contract — they are an implementation detail. The binary record layout is documented in the header comment of `mam4_dump_state.F90`.
